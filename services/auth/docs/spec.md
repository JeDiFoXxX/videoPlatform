# Cервис для авторизации auth — Компонент videoPlatform

## Цель

Микросервис представляет собой центральный узел безопасности на базе JWT и REST API. 
Он предназначен для регистрации, аутентификации пользователей, управления ролями 
и выдачи цифровых токенов доступа для всех компонентов платформы videoPlatform. Сервис работает с 
постоянным хранением данных и управляет собственной базой данных. Вся информация о пользователях, 
хэшах паролей, правах доступа, а также черный список отозванных токенов хранится в ней.

## Стек

- Java 21
- Maven
- Spring Boot
- Docker
- PostgreSQL
- BCrypt (хэширование паролей)
- JWT (Json Web Token)

## Сценарий работы

1. Регистрация и управление пользователями
   Администраторы (ADMIN): Аккаунт для управления системой. 
   Первая учетная запись администратора создается вручную при развертывании 
   БД (через миграции/скрипты инициализации), последующие админы могут создаваться 
   уже существующими админами.
   Ученики (STUDENT): Регистрируются на платформе самостоятельно. 
   При регистрации роль выбрать нельзя — сервер по умолчанию жестко присваивает им статус ученика.
   Учителя (TEACHER): Самостоятельная регистрация заблокирована. 
   Аккаунт учителя может создать только действующий администратор платформы через 
   свою закрытую панель управления.
   Требования к входным данным:
   Логин:
   - Исключительно латинские буквы и цифры.
   - Длина: до 20 символов.
   Пароль:
   - Разрешены латинские буквы, цифры и специальные символы.
   - Длина: от 10 до 20 символов.
   - Обязательное наличие: как минимум одной строчной буквы.О
   - Обязательное наличие: как минимум одной заглавной буквы.
   - Обязательное наличие: как минимум одной цифры.
   - Обязательное наличие: не менее 3 специальных символов (например: !@#$%^&*).

2. Аутентификация и выпуск токенов
   Сервер проверяет учетные данные по БД PostgreSQL, 
   сверяет хэш пароля с помощью BCrypt и генерирует пару токенов: Access Token (короткоживущий) и 
   Refresh Token (долгоживущий).

3. Структура Access Token
   Access Token (JWT) зашиваются публичные данные пользователя: его ID, имя и 
   роль (ADMIN, TEACHER, STUDENT). Токен подписывается секретным ключом сервера.

4. Использование токенов клиентом
   Клиент сохраняет полученные токены и использует Access Token 
   для авторизации всех последующих запросов, включая передачу токена в параметрах при открытии 
   WebSocket-соединения с внешним signaling-сервисом.

5. Валидация запросов
   При каждом запросе к защищенным эндпоинтам сервис локально проверяет криптографическую 
   подпись JWT-токена. Если подпись верна и токен не просрочен, система сразу считывает ID и 
   роль пользователя из самого токена без единого запроса к базе данных.

6. Обновление сессии (Refresh)
   Когда срок действия Access Token истекает (каждые 15 минут), клиент автоматически и 
   незаметно для пользователя отправляет запрос на обновление, передавая сохраненный Refresh Token. 
   Сервер проверяет его валидность в БД PostgreSQL и 
   в случае успеха генерирует и возвращает новый Access Token.

7. Выход из системы (Logout)
   При выходе из системы сервер удаляет Refresh Token из БД PostgreSQL, 
   а клиент полностью стирает обе заготовки токенов из своей локальной памяти. 
   С этого момента зайти в систему под этой сессией или обновить её невозможно.

## API и Точки подключения

### WS `api/v1/auth`
Основной набор HTTP эндпоинтов для управления доступом.

## Аутентификация и Сессии (Клиент -> Сервер)

- POST /api/v1/auth/register — Публичная регистрация для учеников. 
  Запрос принимает логин/пароль и автоматически создает пользователя с ролью STUDENT.
- POST /api/v1/auth/register/teacher — Защищенный эндпоинт создания учителя. 
  Доступен только пользователям с ролью ADMIN. Администратор сам передает данные нового преподавателя.
- POST /api/v1/auth/login — Вход в систему и генерация токенов (expires_in: 900).
- POST /api/v1/auth/refresh — Продление сессии с помощью Refresh-токена.
- POST /api/v1/auth/logout — Мгновенный отзыв токенов и занесение Access Token в черный список PostgreSQL.


### Пример сообщения для входа в систему (Клиент -> Сервер):
```json
{
   "type": "login",
   "login": "teacher123@videoplatform.ru",
   "password": "strong_password_hash"
}
```

   "refresh_token": "rfr_987654321_xyz",
