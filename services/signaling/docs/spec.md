# Сигнальный сервис signaling — Компонент videoPlatform

## Цель

Микросервис представляет собой WebRTC сервер на базе WebSockets.
Он предназначен для управления статусами присутствия («онлайн»), коммутации вызовов 
и установления прямого p2p-соединения (аудио/видеосвязь, трансляция экрана) 
между участниками образовательной платформы videoPlatform.

Сервис работает без постоянного хранения данных и не имеет собственной базы данных. 
Вся информация о подключениях и звонках хранится в оперативной памяти. 
Для проверки прав пользователей он взаимодействует с auth сервисом.

## Стек

- Java 21
- Maven
- Spring Boot
- Docker
- WebRTC / WebSockets

## Сценарий работы

1. Пользователь запрашивает токен (JWT) у сервиса auth.

2. Пользователь открывает WebSocket соединение с signaling сервисом, передавая свой JWT-токен. 
   Сервер валидирует токен, сохраняет сессию в оперативную память приложения 
   и рассылает другим участникам (например, учителю) уведомление о том, что пользователь в сети.

3. Учитель видит список учеников «онлайн» и отправляет запрос на вызов конкретного ученика. 
   Сервер генерирует уникальный ID комнаты (roomId) и пересылает ученику сигнал о входящем вызове.

4. Ученик нажимает кнопку «Принять». Его браузер отправляет на сервер согласие, 
   после чего сервер связывает их в созданной виртуальной комнате.

5. Только после принятия вызова браузеры участников начинают пересылать друг другу 
   через WebSocket технические сетевые адреса (SDP и ICE) для настройки медиа-потоков.

6. Браузеры связываются напрямую (peer-to-peer), и начинается трансляция видео/аудио в обход сервера.

7. При завершении звонка (leave_room) сервер удаляет комнату из памяти. 
   При закрытии вкладки браузера сервер удаляет пользователя из списка «онлайн».

## Типы WebSocket сообщений (Signaling Payload)

Участники и сервер общаются через единый WebSocket-эндпоинт, обмениваясь JSON-сообщениями следующих типов:

### Системные и Статусные (Сервер <-> Клиент)
- `user_online` — уведомление от сервера о том, что пользователь зашел в сеть.
- `user_offline` — уведомление от сервера о том, что пользователь отключился.

### Коммутация звонка (Клиент -> Сервер -> Клиент)
- `make_call` — запрос от учителя на вызов ученика по его ID.
- `incoming_call` — уведомление от сервера ученику о входящем вызове.
- `accept_call` — согласие ученика на принятие вызова и старт созвона.
- `reject_call` — отклонение вызова (учеником или автоматический отказ, если занят).

### Настройка WebRTC (Клиент -> Сервер -> Клиент)
- `sdp_offer` — передача локального описания медиа-потока (аудио/видео кодеки) от инициатора.
- `sdp_answer` — ответное описание медиа-потока от принимающей стороны.
- `ice_candidate` — передача сетевых параметров (IP/порты) для обхода NAT и поиска прямого пути.
- `leave_room` — завершение созвона и выход из комнаты.

## API и Точки подключения

### WS `/ws/signaling?token=BASE64_JWT_TOKEN`
Основной эндпоинт для установления постоянного двустороннего соединения.

### Пример сообщения для инициации звонка (Учитель -> Сервер):
```json
{
  "type": "make_call",
  "sender_id": "teacher-123",
  "target_id": "student-321"
}
```

### Пример сообщения для обмена SDP (Клиент -> Сервер -> Клиент):
```json
{
  "type": "sdp_offer",
  "room_id": "call-uuid-111",
  "sender_id": "teacher-123",
  "target_id": "student-321",
  "payload": {
    "audio": true,
    "video": true,
    "sdp": "v=0\no=- 45672... (строка WebRTC SDP)"
  }
}
```
